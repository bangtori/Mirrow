이 문서는 AI 에이전트(Claude Code, Codex, ChatGPT 등)가 이 저장소에서 작업할 때 따라야 하는 공통 작업 원칙을 정의한다.

---

# 1. 작업 범위 원칙

요청된 범위 내에서만 작업한다.

- 요청 범위 외 수정 금지
- 관련 없는 리팩토링 금지
- 관련 없는 파일 수정 금지
- 발견한 문제는 임의 수정하지 말고 보고
- 기능 수정과 구조 개선은 분리

예시:

- 이벤트 로그 추가 작업 중 디자인 수정 금지
- SEO 작업 중 비즈니스 로직 수정 금지
- 버그 수정 중 폴더 구조 변경 금지

---

# 2. 설계 우선 원칙

기본적으로 즉시 구현하지 않는다.

사용자가 아래와 같은 의도를 가진 경우:

- 어떻게 할지 고민
- 구조 설계
- 방향성 검토
- 선택지 비교
- 아키텍처 논의
- 리팩토링 검토

코드 수정 없이 아래만 수행한다.

- 문제 정의
- 선택지 정리
- 장단점 비교
- 영향 범위 분석
- 권장안 제시

구현은 사용자의 명시적 요청 이후 진행한다.

예시:

- "어떻게 하는 게 좋을까?" → 설계
- "한번 생각해보자" → 설계
- "진행해" → 구현
- "수정해" → 구현

---

# 3. 문서 작성 규칙

모든 프로젝트 문서는 기본적으로 한글로 작성한다.

대상:

- README
- CHANGELOG
- GitHub Issue
- GitHub Release
- 마일스톤
- 기술 문서
- 작업 기록

예외:

- 코드
- 변수명
- 함수명
- 타입명
- Conventional Commit

사용자가 요청하지 않은 영문 문서를 생성하지 않는다.

---

# 4. Git / Release 규칙

커밋 메시지는 Conventional Commit 형식을 사용한다.

형식:

type: 작업 내용

예시:

feat: 이벤트 로그 추가
fix: 응답 페이지 오류 수정
refactor: 결과 계산 로직 정리
docs: changelog 업데이트
chore: v1.0.2 릴리즈 준비

커밋 메시지는 한글로 작성한다.

본문은 선택사항이다.

예시:

feat: 계산 기능 추가

- 더하기 기능 추가
- 빼기 기능 추가

릴리즈 작업 시:

1. package.json 버전 업데이트
2. Footer 버전 업데이트
3. CHANGELOG 반영
4. Git Tag 생성

CHANGELOG에는 실제 배포된 내용만 기록한다.

# 5. 작업 결과 보고 규칙

작업 완료 후 반드시 아래 내용을 보고한다.

## 변경 파일

수정된 파일 목록

## 변경 내용

무엇을 변경했는지 요약

## 검증 결과

실행한 테스트 및 결과

## 남은 작업

추가로 필요한 작업

## 주의 사항

위험 요소 또는 확인이 필요한 내용

---

# 6. 금지 행동

다음 행동은 수행하지 않는다.

- 추측으로 코드 수정
- 관련 없는 리팩토링
- 불필요한 의존성 추가
- 사용하지 않는 코드 대량 생성
- 하드코딩된 환경 변수 추가
- API Key 커밋
- Secret 노출
- 테스트 없이 릴리즈 처리
- 배포되지 않은 기능을 Release Note에 포함
- 사용자 요청 없는 대규모 구조 변경

확신이 없는 경우 수정하지 말고 먼저 보고한다.

---

# 7. 기존 패턴 우선 원칙

프로젝트의 기존 구현 방식을 우선 따른다.

새로운 패턴 도입 전:

1. 유사 기능 존재 여부 확인
2. 기존 패턴 확인
3. 일관성 유지 가능 여부 확인

새로운 패턴이 필요한 경우:

- 기존 방식의 한계
- 변경 이유
- 기대 효과

를 설명한다.

개인 취향보다 프로젝트 일관성을 우선한다.
