이 문서는 AI 에이전트(Claude Code, Codex, ChatGPT 등)가 이 저장소에서 작업할 때 따라야 하는 공통 작업 원칙을 정의한다.

---

# 1. 작업 범위 원칙

요청된 범위 내에서만 작업한다.

- 요청 범위 외 수정 금지
- 관련 없는 리팩토링 금지
- 관련 없는 파일 수정 금지
- 발견한 문제는 임의 수정하지 말고 보고
- 기능 수정과 구조 개선은 분리

예시:

- 이벤트 로그 추가 작업 중 디자인 수정 금지
- SEO 작업 중 비즈니스 로직 수정 금지
- 버그 수정 중 폴더 구조 변경 금지

---

# 2. 설계 우선 원칙

기본적으로 즉시 구현하지 않는다.

사용자가 아래와 같은 의도를 가진 경우:

- 어떻게 할지 고민
- 구조 설계
- 방향성 검토
- 선택지 비교
- 아키텍처 논의
- 리팩토링 검토

코드 수정 없이 아래만 수행한다.

- 문제 정의
- 선택지 정리
- 장단점 비교
- 영향 범위 분석
- 권장안 제시

구현은 사용자의 명시적 요청 이후 진행한다.

예시:

- "어떻게 하는 게 좋을까?" → 설계
- "한번 생각해보자" → 설계
- "진행해" → 구현
- "수정해" → 구현

# 3. 문서 작성 규칙

모든 프로젝트 문서는 기본적으로 한글로 작성한다.

대상:

- README
- CHANGELOG
- GitHub Issue
- GitHub Release
- 마일스톤
- 기술 문서
- 작업 기록

예외:

- 코드
- 변수명
- 함수명
- 타입명
- Conventional Commit

사용자가 요청하지 않은 영문 문서를 생성하지 않는다.

# 4. Git / Release 규칙

## Git 메시지 작성 규칙

커밋 메시지는 Conventional Commit 형식을 사용한다.

형식:

type: 작업 내용

예시:

feat: 이벤트 로그 추가
fix: 응답 페이지 오류 수정
refactor: 결과 계산 로직 정리
docs: changelog 업데이트
chore: v1.0.2 릴리즈 준비

커밋 메시지는 한글로 작성한다.

본문은 선택사항이다.

예시:

feat: 계산 기능 추가

- 더하기 기능 추가
- 빼기 기능 추가

## 릴리즈 작업

### PR 생성 전

릴리즈 준비 요청을 받은 경우 아래 항목 완료 여부를 사용자에게 안내한다.

- 관련 이슈 완료
- 마일스톤 상태 확인
- 로컬 검증 완료
- Preview 검증 완료

AI는 위 항목을 직접 검증할 수 없다고 가정한다.

사용자가 완료되었다고 확인한 경우 이를 신뢰하고 다음 작업을 진행한다.

사용자가 명시적으로 릴리즈 준비를 요청한 경우, 위 항목이 완료되었다고 가정하고 작업을 진행할 수 있다.

확인 항목은 작업 중단 사유가 아니라 안내 사항으로 취급한다.

다음 작업을 수행한다.

- package.json 버전 업데이트
- Footer 버전 업데이트
- CHANGELOG 업데이트
- 릴리즈 준비 커밋 생성
  - 커밋 메시지 예시: chore: v1.1.0 릴리즈 준비

작업 완료 후:

1. 변경 사항을 요약하여 보고한다.
2. 사용자가 PR을 생성하고 main에 머지하도록 안내한다.
3. Git Tag 생성은 수행하지 않는다.
4. Git Tag는 PR 머지 및 운영 배포 완료 이후 진행한다.
5. 사용자가 배포 완료를 알리면 PR 머지 이후 절차를 진행한다.

---

### PR 머지 이후

AI는 아래 항목을 직접 검증할 수 없다고 가정한다.

- Production 배포 확인
- 운영 환경 검증
- 주요 사용자 흐름 테스트

위 항목은 사용자가 수행한다.

사용자가 배포 및 검증이 완료되었다고 확인한 경우 이를 신뢰하고 다음 작업을 진행한다.

사용자가 명시적으로 태그 생성 또는 릴리즈 마무리를 요청한 경우, 위 항목이 완료되었다고 가정하고 작업을 진행할 수 있다.

다음 작업을 수행할 수 있다.

- Git Tag 생성
- Git Tag Push
- main → dev 동기화

Git Tag는 운영 배포가 완료된 main 커밋에 생성한다.
태그 작업 전 main 브랜치의 최신 상태를 반영한다.

예시:

```sh
git checkout main
git pull origin main

git tag v1.1.0
git push origin v1.1.0
```

태그 작업 후 main과 dev의 상태를 확인한다.
main에만 존재하는 커밋이 있는 경우 dev 브랜치에 동기화한다.

예시:

- main에서 진행한 핫픽스
- main에서 직접 수행한 긴급 수정
  일반 릴리즈 과정에서 dev → main으로 배포된 경우에는 별도 동기화가 필요하지 않을 수 있다.

```sh
git checkout dev
git merge main
git push origin dev
```

# 6. 금지 행동

다음 행동은 수행하지 않는다.

- 추측으로 코드 수정
- 관련 없는 리팩토링
- 불필요한 의존성 추가
- 사용하지 않는 코드 대량 생성
- 하드코딩된 환경 변수 추가
- API Key 커밋
- Secret 노출
- 테스트 없이 릴리즈 처리
- 배포되지 않은 기능을 Release Note에 포함
- 사용자 요청 없는 대규모 구조 변경

확신이 없는 경우 수정하지 말고 먼저 보고한다.

---

# 7. 기존 패턴 우선 원칙

프로젝트의 기존 구현 방식을 우선 따른다.

새로운 패턴 도입 전:

1. 유사 기능 존재 여부 확인
2. 기존 패턴 확인
3. 일관성 유지 가능 여부 확인

새로운 패턴이 필요한 경우:

- 기존 방식의 한계
- 변경 이유
- 기대 효과

를 설명한다.

개인 취향보다 프로젝트 일관성을 우선한다.
